# 菜单打开成本刷新与额度告警图标设计

## 目标

解决两个实机问题：打开菜单不再触发昂贵的全量会话日志重扫；额度进入告警或耗尽状态时继续显示新的额度进度图标，不再退回闪电或三角形系统符号。

## 方案比较

### 方案 A：按刷新来源分流（采用）

- 菜单打开仍立即刷新 OAuth 额度，但本地成本只读取现有 ledger/缓存，不主动刷新 session cache。
- 用户点击刷新按钮时保留深度刷新能力，允许扫描变动会话并更新 ledger。
- 改动小，保留手动纠错入口，也不会让每次打开菜单重新解析大型日志。

### 方案 B：菜单打开完全不刷新成本

- 性能最好，但 Today 成本在长时间不手动刷新时可能明显滞后。
- 不采用，因为现有 ledger 快速归约可以低成本提供较新的结果。

### 方案 C：本轮直接实现 JSONL 追加量增量解析

- 长期性能最佳，但需要持久化字节偏移、解析状态和截断恢复，改动面明显更大。
- 留作后续独立优化，不作为修复菜单打开卡顿的前置条件。

## 刷新数据流

- 新增明确的刷新来源：`menuOpen` 与 `manual`。
- `menuOpen`：OAuth 强制刷新；成本刷新使用 `refreshSessionCache: false`，优先从持久化 ledger 归约。
- `manual`：OAuth 与成本均按用户意图深度刷新，成本使用 `refreshSessionCache: true`。
- `TokenStore` 现有的空数据/不完整数据兜底仍保留；只有缓存不可用时才允许恢复性扫描，不再由普通菜单打开无条件触发。

## 图标与颜色

- 有真实额度窗口时，正常、低额度、短窗口耗尽和长窗口耗尽都使用 `MenuBarUsageIconRenderer`。
- 低额度使用橙色，耗尽的长窗口使用红色；进度本身继续表达剩余/已用比例。
- 更新可用、账号封禁、无额度数据和兼容 Provider 继续使用明确系统符号。
- VoiceOver 仍报告所有实际窗口和告警语义。

## 测试

- 纯策略测试锁定菜单打开不刷新 session cache、手动刷新会刷新 session cache。
- 展示测试锁定低额度与额度耗尽都生成 usage-bars 图标，并得到橙色/红色强调。
- 现有更新、封禁和 Provider 系统符号优先级测试保持通过。
- 完整 XCTest 后重装本地 App；实机打开菜单，确认 CPU 不因菜单打开升到单核满载，且当前低额度账号显示新额度图标。
